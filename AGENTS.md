# AGENTS.md

## 项目定位

这是一个基于 Node.js/TypeScript 的 LLM 聊天代理服务，当前可接入 OneBot，也支持仅通过 WebUI 操作，核心能力包括：

- OneBot 事件接入与消息发送
- 会话编排、历史压缩、回复门控与发送队列
- persona、用户资料与记忆持久化
- unified shell tools (shell_run, shell_interact, etc.) with timeout-to-background and PTY support
- 定时任务、白名单、请求审批与内部 API
- WebUI（Vue 3 + Tailwind）：会话监控、配置编辑器、数据编辑器

## 开发约束

### 0. 代码质量第一原则

本项目的第一要求是不堆屎山，始终优先保持结构清晰、功能分割明确、代码可读、可维护。

这意味着默认应遵循以下原则：

- 优先做职责清晰的拆分，而不是把新逻辑继续堆进已有混杂模块
- 优先消除隐式耦合、散落分支和特例硬编码，而不是继续在原位置打补丁
- 优先让配置、运行时能力、数据结构、调用链边界保持一致
- 优先选择长期可维护的清晰结构，而不是“先凑合能跑”的短期方案
- 发现现有设计已经开始堆积复杂度时，应主动整理结构，而不是容忍复杂度继续扩散

### 1. 默认不保留向后兼容代码

当项目功能、配置结构、数据结构、模块边界发生变更时，默认直接收敛到新实现，不为了兼容旧行为额外保留过渡层，除非任务明确要求兼容旧版本。

这意味着默认可以直接做以下事情：

- 删除旧分支、旧参数、旧字段、旧工具别名
- 删除仅为兼容旧配置/旧数据格式存在的适配逻辑
- 删除废弃的迁移兜底、兼容性 fallback、双写/双读代码
- 直接更新调用方、测试、文档，使其以当前设计为准

### 2. 变更优先级

实现功能变更时，优先保证以下几点：

- 新结构清晰且单一
- 代码路径可维护，不堆叠兼容判断
- 测试覆盖当前行为，而不是继续验证已废弃行为
- README、配置示例、测试数据与实现保持一致
- 接受必要的大规模重构，但重构后结构必须更整洁、更单一，而不是把复杂度转移到别处

### 3. 兼容性例外

只有在以下情况，才保留或新增兼容层：

- 任务说明明确要求向后兼容
- 明确要求保留旧配置或旧数据迁移能力
- 明确要求保留外部接口协议兼容性

如果没有这些明确要求，就不要主动添加兼容代码。

## 修改建议

- 修改功能时，优先检查 `src/` 中的真实调用链，而不是围绕旧行为叠加补丁
- 修改配置或持久化结构时，同时更新 `README.md`、`config/*.yml` 示例和相关测试
- 做破坏兼容性的重构时，需要同步更新或删除 `config` 中已经过时的旧配置结构，不要保留失效字段或旧写法
- 修改行为后，删除无效旧实现，避免留下“以后也许会用”的死代码
- 如果修改了会话消息清单、发送队列、transcript 类型、SSE 会话流或其他会直接影响 WebUI `Sessions` 页展示语义的后端逻辑，必须同步更新前端会话页与相关文档，不要只改后端数据链路
- `webui/` 前端默认优先使用 Tailwind utility class；只有全局主题变量、浏览器基础样式和少量 utility 难以表达的场景才写 CSS
- `dist/` 和 `node_modules/` 属于产物或依赖，不作为源码修改目标
- 准备提交 commit 前，必须至少跑通 `npm run typecheck:all` 和 `npm run test`

## 文档与 Agent 工作文件约定

### 1. `docs/` 只放长期保留的项目知识

`docs/` 中的内容默认应面向未来维护者，而不是面向单次执行过程。只有在以下信息具备长期价值时，才应写入 `docs/`：

- 当前系统结构、边界与约束
- 已采纳的设计决策与 trade-off
- 尚未实现但需要长期保留的 backlog / proposal
- 运维、迁移、配置说明等稳定参考材料

### 2. agent 过程文档默认不直接进入 `docs/`

agent 在实现前后产生的 spec、plan、review、执行 checklist、临时设计稿、问题拆解记录等，默认属于过程产物，而不是正式项目文档。

这些文件默认应写入仓库根目录下的 `/.superpowers/` 或其他明确约定且已加入 `.gitignore` 的本地工作目录，而不是直接写入 `docs/`。

除非用户明确要求提交这些过程文档，否则不要把它们作为正式仓库内容纳入版本控制。

### 3. 需要持久化时，先提炼再落正式文档

如果一次 agent 工作中产出了确实需要长期保留的信息，不应直接把原始 spec/plan/checklist 原样放进 `docs/`，而应先提炼为“面向维护者的结果文档”后再写入正式位置。

默认按以下口径归档：

- 架构与现状说明：写入 `docs/` 下对应主题文档
- 已采纳的设计决策：写入设计决策/方案文档
- 未完成但值得长期追踪的功能方案：写入 backlog / proposal 文档
- 团队协作规则与 agent 工作边界：写入 `AGENTS.md`、`README.md` 或相关开发文档

### 4. 历史过程文档应提炼后删除，不长期保留

仓库里已经存在的 agent 过程文档，如果其中包含需要长期保留的信息，应先抽取、提炼并写入正式文档位置，再删除原始过程文件。

默认不要为了“以后也许有用”继续在仓库里堆积历史 spec、plan、checklist 或 review 记录。

### 5. 临时备份与执行痕迹不是正式文档

以下内容默认不属于正式项目文档：

- `*.main-untracked-backup.md` 这类临时备份
- 只用于本次修改的 step-by-step 执行清单
- 纯 review 草稿、过程备注、模型中间产物

这类文件应留在本地工作目录或直接清理，不应作为正式文档长期保留在仓库中。

## 开发工作流约定

- 默认在独立 `git worktree` 中开发，不直接在主目录工作副本上做日常改动。
- 如果主目录正在运行 watch/dev 进程并监控主目录文件，worktree 中的源码修改默认不应触发主目录 dev 自动重启。
- 主目录工作副本默认视为稳定运行副本；默认应优先在开发用 worktree 内完成代码修改、typecheck、test 与本地联调，确认大体可用后再决定是否合并到主目录验证。
- 从 worktree 合并回 `main` 时，默认存在两种受用户指令控制的路径：
  - 直接整理为最终正式提交后合并到 `main`，合并后不再在 `main` 额外做本地测试
  - 先整理为 `wip` 提交合并到 `main` 做联调或验收，待主目录验证通过后，再把本次历史整理为最终正式提交
- 如果用户没有明确指定合并策略，默认采用“先在 worktree 内完成验证，再停下来等待用户确认是否合并以及采用哪条路径”的做法，而不是自动假设必须先进 `main` 做测试。
- 开发 worktree 中的改动在完成实现并通过本地验证后，默认先停下来等待用户确认；除非任务明确要求立即合并、提交、push 或清理，否则不要在完成开发后直接合并回 `main`。
- 需要合并回 `main` 时，默认先由用户明确确认再执行；不要把“实现完成”自动等同于“可以直接 merge”。
- 合并回 `main` 做测试时，默认视为“临时验收态”而不是最终历史；在用户明确确认“没问题”之前，不要默认立即删除本次开发用分支或 worktree。
- 用户确认 `main` 上测试/验收无问题后，默认应把本次开发整理成正式提交后再作为最终保留历史；如果当前 `main` 上是本地未 push 的 `wip` 历史，可以重写为更干净的正式提交；如果相关历史已经 push，则不要擅自重写，必须先征求用户确认。
- 如果主目录验收期间发现问题，默认优先回开发 worktree 修正，并更新或替换本次 `wip` 历史后再回到主目录继续验收；不要在 `main` 上连续堆叠多个临时修补 commit，除非用户明确要求保留这些中间提交。
- 用户确认 `main` 上测试/验收无问题后，应清理本次开发已不再需要的 worktree 和已完成合并的分支；如果用户明确要求保留，则按用户要求保留。
- 不要依赖跨 worktree 搬运未提交改动到主目录进行联调；主目录验证默认基于已提交的开发切片。
- 需要联调新代码时，优先选择以下方式之一：
  - 在开发用 worktree 内单独启动 dev / build / test
  - 将开发分支合并到主目录当前分支后，在主目录运行 dev 验证
- 如果需要在 worktree 内启动可长期运行的本地服务做联调，模型可以从主目录工作副本复制一份完整的公共配置与 `dev` 实例配置到该 worktree，再派生一个专用实例配置供本次联调使用。
- 该 worktree 联调实例默认应至少满足以下要求：
  - `appName` 与实例名明确区分于 `dev`
  - `dataDir` 使用 worktree 专属目录，避免与 `dev` 共享运行数据
  - `internalApi.port` 与 `internalApi.webui.port` 改成不会与现有 `dev` 冲突的端口
- 模型在完成上述复制和调整后，应明确告诉用户本次 worktree 联调实例名，并给出准确的启动命令；默认使用：
  - `CONFIG_INSTANCE=<wt-instance-name> npm run dev`
- 如果测试、脚本或 dev 进程会写共享外部资源（固定端口、同一 data 目录、systemd 服务状态、仓库外固定文件等），必须主动检查是否会与主目录运行实例互相影响，不要假设 worktree 能隔离这类副作用。
- 如果模型需要使用 Superpowers 的浏览器演示/可视化展示能力，且当前存在可用的开发 worktree，默认优先在该 worktree 目录中启动演示服务，并监听 `0.0.0.0:23380`。
- 这样做的原因是浏览器演示服务会产生 `.pid` 等本地运行痕迹；这类运行时垃圾文件应优先留在 worktree 中，而不是污染主目录稳定副本。
- 多个 worktree 并行开发时，尽量按目录或职责切分改动范围；最终冲突合并应由了解主线结构的人统一处理，不要让多个并行分支各自随意解决同一组结构冲突。

## Prompt 语言约定

- 与用户的默认沟通语言为中文；除非用户明确要求使用其他语言，否则分析、说明、计划、总结等都应使用中文。
- 本项目中的所有 prompt 默认使用中文编写，包括 system prompt、developer prompt、tool prompt、总结提示词、压缩提示词、persona 相关提示词及其他内部提示文本
- 除非任务明确要求使用其他语言，或外部接口协议明确要求特定语言，否则不要使用英文或中英混写 prompt
- 修改或新增 prompt 时，同时检查相关测试、示例配置与文档描述是否仍与中文 prompt 约定一致

如果仓库中的现状与本文档描述不一致，以实际代码结构为准，并同步更新本文件。
