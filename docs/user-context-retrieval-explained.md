# 用户上下文自动回忆机制说明

这份文档面向普通用户，解释系统如何“记住”和“想起”与你有关的历史上下文。

## 它解决什么问题

聊天助手经常会遇到一个问题：你之前明明说过某些偏好、项目背景或任务细节，但新一轮聊天时，模型不一定能自动想起来。

传统做法是让模型主动把重要内容写进“记忆”。这个方式有两个缺点：

- 模型不一定每次都会主动写。
- 写进去的内容可能太多、太杂，后面每次都塞进 prompt 会浪费 token。

新的用户上下文检索机制会更像“自动整理聊天资料”：

1. 聊天发生后，系统把历史保存下来。
2. 系统把历史整理成可检索的小片段、摘要和长期事实。
3. 每次你发新消息前，系统先自动查找与你当前问题相关的旧信息。
4. 只把少量最相关的信息交给模型参考。

## 它不是简单地把所有历史都塞给模型

系统不会把全部聊天记录直接发给模型。

原因很简单：

- 历史太多会浪费 token。
- 太多无关内容会干扰回答。
- 旧细节可能已经过期。
- 有些内容只适合保存，不适合每次都注入。

所以系统会做筛选：当前问题需要什么，就尽量只取相关的几条。

## 系统把信息分成几层

可以把它理解成四个抽屉。

### 1. 原始记录

这是最完整的聊天原文。

用途：

- 以后需要时可以回溯。
- 可以重新整理摘要。
- 换 embedding 模型时可以重建索引。

原始记录默认不会直接发给模型。

### 2. 近期片段

这是从近期聊天里切出来的小段内容。

适合保存：

- 最近正在做的任务。
- 路径、命令、配置、错误信息。
- 临时要求。
- 还不确定是否长期有效的信息。

例子：

- “这次先在新 worktree 里做 POC。”
- “NAS 清理默认先 dry-run。”
- “刚才测试失败在某个 memory 测试。”

这类信息近期有用，但不一定永远重要。

### 3. 历史摘要

当近期片段变旧后，系统会把一批片段压缩成摘要。

适合保存：

- 一段会话的结论。
- 一个任务之前做到了哪一步。
- 旧聊天的大概背景。

例子：

- “之前评估过 Orama 作为轻量本地检索索引，保留以后迁移到 LanceDB 或 SQLite 的空间。”

摘要比原始片段短，适合长期保存和检索。

### 4. 长期事实

这是最稳定、最应该长期记住的信息。

适合保存：

- 你的长期偏好。
- 你明确说“以后都这样”的要求。
- 项目长期规则。
- 被多次确认的事实。

例子：

- “用户默认希望中文回答。”
- “用户偏好先给结论，再展开理由。”
- “这个项目默认在独立 worktree 开发。”

系统会很保守地把内容提升为长期事实。临时要求不会轻易变成长期记忆。

## 系统如何判断该放哪一层

所有消息都会先进入原始记录。

之后系统会判断：

- 近期可能有用、细节重要：放入近期片段。
- 旧了但仍有背景价值：整理成历史摘要。
- 长期稳定、跨会话有用、表达明确：保存为长期事实。
- 只是一次性要求：不会保存成长期事实。

如果你说了类似这些话，系统更可能认为它是长期事实：

- “以后默认这样。”
- “记住这个。”
- “我喜欢这种方式。”
- “我不喜欢那种方式。”
- “每次都先做这个。”
- “不要再这样做。”

如果你只是说“这次先这样”，系统会把它当作临时上下文。

## 每次聊天前发生了什么

你发出新消息后，系统会先做一个检索步骤：

1. 读取当前用户身份。
2. 根据你的新消息生成一个查询。
3. 只在与你有关的历史里查找。
4. 同时用关键词和语义相似度查找。
5. 按相关性、时间新近程度、重要性排序。
6. 去掉重复或太相似的内容。
7. 挑出少量最有用的信息放进 prompt。

模型最终看到的不是完整历史，而是类似这样的简短上下文：

```text
- [长期事实] 用户希望默认中文回答。
- [历史摘要] 之前讨论过 Orama 版用户上下文检索 POC。
- [近期片段] 当前 POC 位于 poc/user-context-hybrid-retrieval-orama。
```

## 是否会记住其他人的信息

默认不会。

第一阶段只检索“当前触发用户相关”的历史。也就是说，你发消息时，系统默认只找与你有关的上下文。

这条边界很重要，因为聊天机器人可能同时服务多个用户、多个群或多个会话。

## 是否必须永远使用同一个 embedding

不需要永远使用同一个 embedding，但同一个索引里不能混用不同 embedding。

可以这样理解：

- embedding 模型负责把文字变成一串数字。
- 不同模型生成的数字空间不一样。
- 旧模型生成的向量，不能直接和新模型生成的查询向量混在一起比较。

所以系统会记录当前使用的 embedding profile。

切换 embedding 时，系统会：

1. 保留原始文字和摘要。
2. 用新 embedding 模型重新生成向量。
3. 建一个新索引。
4. 确认新索引可用后再切换。
5. 旧索引保留一段时间后清理。

只要原始文字还在，历史不会因为换 embedding 而丢失。

## 历史会不会越积越多

不会无限增长。系统会做自动清理。

清理不是简单删除，而是分阶段处理：

- 近期片段旧了以后，会先被整理成摘要。
- 已经被摘要覆盖的旧片段，会从热检索层移走。
- 重复、长期没用、低重要性的信息会被归档。
- 被你 pin 的内容不会自动清理。
- 你明确删除的内容会进入删除状态，过一段时间后才物理清除。

这样可以保留长期有价值的信息，同时避免历史容量爆炸。

## 你可以手动管理什么

后续 WebUI 或聊天命令可以提供这些能力：

- 查看系统记住了哪些长期事实。
- 查看某个会话生成了哪些摘要。
- 删除某条记忆。
- 删除某个会话相关上下文。
- pin 重要信息，防止自动清理。
- unpin 不再重要的信息。
- 编辑错误的事实或摘要。
- 要求系统忘记某个话题。
- 导出自己的历史上下文。

常见命令可以是：

```text
记住这条
忘记这段
忘记这个会话
忘记关于某个话题的信息
不要把这个会话用于长期记忆
```

## 什么时候它会出错

这个机制不是完美的，可能出现几类问题：

- 召回了相关但已经过期的信息。
- 没召回真正需要的信息。
- 把临时信息误认为长期偏好。
- 摘要时丢掉了某些细节。

为降低这些风险，系统会：

- 保守提升长期事实。
- 保留原始记录用于追溯。
- 支持手动删除和编辑。
- 给重要信息提供 pin。
- 记录每次为什么召回或丢弃某条上下文。

## 一句话总结

这个机制的目标不是让模型“什么都记住”，而是让系统自动整理你的历史聊天，在需要时只拿出少量最相关、最有用、权限正确的信息给模型参考。
